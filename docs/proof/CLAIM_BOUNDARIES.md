# Claim Boundaries

DesignPilot should stay explicit about what the repo can honestly support.

Avoid implying:
- human-level design intelligence
- automated reasoning as a universal capability
- error-free output
- ROI certainty without longitudinal evidence
- blinded external wins when only internal comparisons exist
- production outcome proof when only artifact receipts or reviewer confidence exist

Prefer bounded claims such as:
- structured multi-step orchestration for design work
- route-specific decision support instead of one-size-fits-all prompting
- review-ready artifacts with explicit constraints and next steps
- stronger internal discipline through validation, regression, continuity, and release gating
- comparative improvements on the benchmarked tasks only
