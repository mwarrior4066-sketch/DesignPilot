# AI-to-Designer Response Filter Report

## Scope
Operational source doc for the designer-readable transformation layer.

## Core thesis
Simplification removes detail. Translation remaps technical meaning into a domain the designer can act on.
The response filter must preserve route integrity, constraints, and proof honesty while changing the explanation surface.

## Middleware position
The filter belongs after draft generation and before final validation.

## Response architecture
- executive summary or core verdict
- impact analysis or contextual framing
- constraints and implementation realism
- actionable next steps

## What changes
- jargon density
- scaffolding depth
- framing
- examples
- next-step guidance

## What does not change
- governing route
- required task decisions
- evidence burden
- implementation constraints
- code or strict spec artifacts

## Actionability rule
Every transformed answer should direct the user toward the next useful move at the depth appropriate to the active tier.

## Failure patterns
- technically correct but unreadable for the current tier
- easy to read but stripped of the real constraint
- no next-step direction
- bloated scaffolding that adds no decision value
