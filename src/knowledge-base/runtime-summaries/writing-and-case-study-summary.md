# Writing And Case Study Summary Runtime Summary

## Decision rules
- identify the writing mode first: case study, audit, rebuild plan, expand proposal, UX copy, or documentation
- lead with the user task, problem, or key point
- use short sentences and active voice
- in case studies, keep the backbone visible: problem → process → solution → outcome
- tie design claims to rationale and evidence instead of adjectives
- annotate visuals so they support the narrative instead of floating as artifacts
- do not mix portfolio, documentation, and UX microcopy conventions blindly

## Failure traps
- writing a rebuild answer like an audit
- writing a case study like marketing copy
- error messages with no next step
- design claims with no evidence
- buried benefits, missing problem framing, or outcomes hidden at the end
- screenshots or artifacts with no caption or narrative role

## Escalate when
- the runtime summary lacks exact thresholds, standards wording, or measurable criteria
- two summaries or skills conflict and the governing rule is unclear
- the validator flags missing evidence classes or insufficient evidence
- the task is high-stakes, standards-grade, or audit-sensitive
- the user explicitly asks for deep rationale or full-source synthesis
