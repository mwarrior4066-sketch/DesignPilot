# PDF and Document Accessibility Expert Operator Pack

## Standards baseline and normative reference set

A “PDF Accessibility Expert” that makes production-level decisions needs to be grounded in normative PDF specifications (what the file *shall* contain), conformance test models (how failures are operationalized), and implementation guidance (how to build/repair PDFs without breaking the semantics). For PDFs, this baseline is anchored in **PDF/UA** (ISO 14289) and the core PDF specifications (**ISO 32000**), which jointly constrain the **tagging/structure tree**, **reading order**, and **Unicode text extraction** pathways that assistive technologies depend on. [\[1\]](https://pdfa.org/resource/iso-14289-pdfua/)

Key normative relationships your expert agent must internalize:

- **PDF/UA-1 (ISO 14289-1:2014)** specifies how to use **ISO 32000-1:2008 (PDF 1.7)** to produce accessible documents. [\[2\]](https://pdfa.org/resource/iso-14289-pdfua/)
- **PDF/UA-2 (ISO 14289-2:2024)** specifies how to use **ISO 32000-2:2020 (PDF 2.0)** for accessible documents, and the PDF Association[\[3\]](https://docs.verapdf.org/validation/) explicitly notes major improvements such as comprehensive attribute/annotation requirements and support for modern Unicode and PDF 2.0 features (e.g., associated files). [\[4\]](https://pdfa.org/iso-14289-2-pdfua-2/)
- **Well-Tagged PDF (WTPDF) 1.0** (a PDF Association specification) operationalizes “well-tagged” requirements in PDF 2.0; in practice, it is closely aligned with PDF/UA-2’s accessibility requirements and provides concrete, implementer-oriented requirements for attributes (tables, artifacts, text replacements, etc.). [\[5\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)
- **ISO/TS 32005:2023** exists because ISO 32000-2 introduced namespaces and child-containment rules for PDF 2.0 structure types, but did not fully specify interaction rules for PDF 1.7 namespace usage inside PDF 2.0. This matters when you encounter mixed tag vocabularies or custom role mapping in PDF 2.0 documents. [\[6\]](https://pdfa.org/resource/iso-32005/)

Conformance testing and what it means for “strict” behavior:

- The **Matterhorn Protocol** is a PDF/UA-1 conformance testing model that enumerates checkpoints and failure conditions (31 checkpoints, 136 failure conditions; with a split between machine-detectable vs human-judgment checks). A strict agent should treat Matterhorn-style failure conditions as “hard stops” for PDF/UA-1 claims and as a template for rule severity classification even when targeting PDF/UA-2. [\[7\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)
- **veraPDF** formalizes “shall” statements from PDF/UA (and WTPDF) into machine-checkable validation profiles; critically, for PDF/UA it only covers machine-verifiable checks, which means a production system must explicitly schedule human or higher-level semantic checks (e.g., is the alt text actually meaningful; is reading order actually correct). [\[8\]](https://docs.verapdf.org/validation/)

Scope boundaries your agent must respect:

- PDF/UA standards define *technical encoding* requirements (tagging, semantics exposure, logical order, Unicode mapping, metadata hooks), but do not regulate many content-quality requirements (e.g., specific contrast ratios, line length, or “plain language” choices). This is explicit in ISO 14289-2’s framing and the PDF Association’s “context and limitations” statements—so your operator pack must separate **file-format conformance** from **content accessibility quality** and implement both as distinct layers. [\[9\]](https://cdn.standards.iteh.ai/samples/82278/61329d7c9eb54e3fbc58a69f6036031d/ISO-14289-2-2024.pdf)

## Operating principles for a strict PDF Accessibility Expert

This operator pack is best modeled as a rule-driven expert system with explicit **precedence**, **invariants**, and **stop conditions** so it can make defensible production decisions.

**Rule precedence (highest to lowest):**  
(1) ISO/PDF normative requirements (PDF/UA, ISO 32000) → (2) WTPDF/ISO/TS 32005 containment rules (for PDF 2.0 tagging correctness) → (3) Matterhorn checkpoints/failure conditions (test operationalization) → (4) PDF Association technical best practices (e.g., Tagged PDF Best Practice Guide: Syntax) → (5) Tooling guidance (Acrobat, PAC, remediation tool UX) for implementation constraints and known pitfalls. [\[10\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)

**Core invariants (treat as “never violate”):**

- **Everything is accounted for:** every piece of page content must be either (a) included as real content in the logical structure, (b) explicitly identified as an artifact, or (c) removed from the logical structure representation exposed to AT (not merely “left floating”). [\[11\]](https://helpx.adobe.com/acrobat/using/editing-document-structure-content-tags.html)
- **Logical order is structure order:** reading order is derived from the logical structure (depth-first traversal of the structure tree), not from paint order or visual layout order, which may be optimized for print/rendering. [\[12\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
- **Semantics are explicit and standard:** the most semantically appropriate tag should be used; when custom structure types exist, role mapping must map them to semantically appropriate standard types rather than misusing headings or generic containers. [\[13\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
- **Text is Unicode-recoverable:** text (including page content, metadata, annotations) must be mapped to Unicode; when a glyph-level mapping is not available or is wrong, replacement mechanisms such as ActualText become critical and must be applied without destroying semantics. [\[14\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
- **Editing must preserve semantics:** edits cannot be “visual-only”; they must preserve and/or update tags, MCIDs/object references, role maps, metadata, and reading order; otherwise the file ceases to be reliably accessible. Tool guidance explicitly warns that editing can corrupt surrounding tag structure and requires post-edit retesting. [\[15\]](https://highways.dot.gov/sites/fhwa.dot.gov/files/had21001.pdf)

**Strictness model (for an operator pack):**

- Interpret **“shall”** as *hard requirement* (Error; must fix or block release). [\[8\]](https://docs.verapdf.org/validation/)
- Interpret **“should”** as *strong default* (Warning; fix unless a justified exception is logged). [\[16\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)
- Interpret **“may”** as optional (Info; implement when it materially improves AT experience or downstream extraction). [\[17\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)

## Topic rulebook for an expert agent

**Topic: PDF/UA structure and tagging**

1.  **Definition.** PDF/UA requires that PDF files expose an accessible logical structure via “Tagged PDF” (structure elements/structure tree) so content is machine-readable, semantically labeled, and consumable in order by assistive technology. PDF/UA-1 is based on PDF 1.7; PDF/UA-2 is based on PDF 2.0 and strengthens requirements (notably attributes/annotations and modern Unicode support). [\[1\]](https://pdfa.org/resource/iso-14289-pdfua/)
2.  **Why it matters for a PDF Accessibility Expert.** Screen readers and other AT depend on the structure tree to identify headings, paragraphs, tables, figures, links, and their relationships; without correct tagging, the document may be readable visually but incoherent or unnavigable nonvisually. Conformance testing models explicitly treat missing/incorrect tagging and semantic misuse as core failure conditions. [\[18\]](https://www.hhs.gov/sites/default/files/pdf-tagging.pdf)
3.  **Default rules.** Tag all real content; ensure tags are semantically appropriate; maintain a single coherent structure hierarchy that reflects the document’s logical organization rather than page layout; ensure custom tags terminate through RoleMap to appropriate standard types; ensure content spanning pages remains one logical element rather than being arbitrarily restarted per page. [\[19\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
4.  **Exception rules.** When the document requires domain-specific structure (e.g., specialized sections, fragments, or reuse-oriented structures), permit custom structure types only with correct RoleMap mapping; avoid mapping custom tags to headings unless they truly are headings, because mis-mapping is a conformance failure pattern. [\[20\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
5.  **Fallback logic.** If tags are missing or clearly unusable, prefer regeneration from the source authoring system when possible; otherwise apply controlled auto-tagging plus a deterministic repair pass that (a) normalizes structure hierarchy, (b) repairs reading order, (c) repairs role mapping, (d) repairs Unicode mapping, and then (e) reruns validation and human sampling checks. Acrobat and multiple government remediation guides treat auto-tagging as a starting point requiring review and manual fixes. [\[21\]](https://helpx.adobe.com/acrobat/using/editing-document-structure-content-tags.html)
6.  **Failure conditions.** Representative hard failures: real content missing from tags; content neither tagged nor artifact; artifacts tagged as real content (or real content marked as artifact); semantically inappropriate structure types; broken role mapping; and for PDF/UA-1, a Suspects flag value of true is explicitly called out as a failure condition in Matterhorn. [\[22\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)
7.  **Measurable thresholds.** Required: 0 instances of untagged real content; 0 instances of misclassified artifact/real content; 0 semantically inappropriate structure type uses (for standard tags); pass machine validation for the targeted profile (PDF/UA-1 or PDF/UA-2; plus WTPDF when targeting UA-2). Advisory: track “human-check required” items separately (e.g., alt text quality). [\[23\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)
8.  **Implementation guidance for an AI operator pack.** Implement a “Structure Tree Contract” object: (a) catalog-level required entries, (b) StructTreeRoot presence and connectivity, (c) RoleMap termination correctness, (d) per-element semantic validation (tag name, allowed children/containment rules per ISO/TS 32005 in PDF 2.0 contexts), (e) marked-content linkage integrity (MCIDs/OBJR mapping), and (f) cross-check with extracted text in reading order. Gate release on contract satisfaction. [\[24\]](https://pdfa.org/resource/iso-32005/)
9.  **Test cases.** A PDF with (a) visible headings but only \<P\> tags; (b) custom tag \<DataTable\> mapped to \<H1\>; (c) text on page that is neither in a tag nor marked artifact; (d) one logical paragraph restarted as new \<P\> on every page break; (e) RoleMap chain that never reaches a standard structure type. [\[25\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)

**Topic: reading order rules**

1.  **Definition.** In accessible PDF, the primary reading order is the logical order established by a depth-first traversal of the structure tree; this compensates for the fact that PDF paint order can be optimized for rendering and may not correspond to human reading order. [\[12\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
2.  **Why it matters for a PDF Accessibility Expert.** Incorrect reading order is one of the fastest ways to make a tagged PDF unusable: users experience “jumping” between columns, headers, footnotes, or page regions; remediation often requires structural changes, not cosmetic ones. Remediation guides emphasize that assistive technology reads in tag order and that tool views can disagree—priority should be given to the tag structure order when conflicts exist. [\[18\]](https://www.hhs.gov/sites/default/files/pdf-tagging.pdf)
3.  **Default rules.** Define order by structure tree; ensure each block element’s children appear in human reading sequence; preserve continuity across page breaks (logical structure is pagination-agnostic); for multipage elements (paragraphs, tables), link all content segments in correct order without restarting the same structure element; set Tab Order to “use document structure” for pages with interactive elements so keyboard navigation aligns with the logical structure. [\[26\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
4.  **Exception rules.** Pagination artifacts (running headers/footers/page numbers) are typically **not part of the main reading order** and should be represented as artifacts rather than interleaved with body content, while still enabling users to orient themselves (e.g., via page labels and pagination artifact typing). Digital signatures are a good example where conforming readers may surface information through a separate UI instead of forcing it into the logical reading order. [\[27\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
5.  **Fallback logic.** When structure order is ambiguous or likely wrong, compute a candidate logical order from layout heuristics (columns, grouping, anchors) but only commit it if it increases consistency with semantic grouping and if verification via screen-reader/structure preview indicates improvement; otherwise flag for human judgment. Validation tools and remediation guides repeatedly emphasize the need for manual verification beyond automated checks. [\[28\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)
6.  **Failure conditions.** Typical failures: reading order that follows visual z-order incorrectly (e.g., right column inserted mid-left column); artifacts embedded within tagged content or tagged content nested inside artifacts; content spanning pages restarted incorrectly; tab order inconsistent with structure order on pages containing annotations/forms. [\[29\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)
7.  **Measurable thresholds.** Required: logical order must be “unambiguous” (no contradictory sequences) and consistent with structure tree traversal; pages with annotations must use structure-based tab order. Recommended internal KPI: “order-correctness sampling”—for each document, sample ≥10 reading transitions (including at least one multipage element and one multi-column or sidebar transition) and require 100% correctness for release. [\[30\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
8.  **Implementation guidance for an AI operator pack.** Build a “ReadingOrderGraph” where nodes are semantic blocks (headings/paragraphs/table regions/figure+caption units) and edges represent intended next reading step; derive initial graph from structure tree; validate with geometric locality constraints and known exceptions (artifacts, marginalia); provide an explicit diffable “reorder plan” (moves in the structure tree) so changes are explainable and reversible. [\[31\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
9.  **Test cases.** Two-column newsletter; sidebar callout box; footnotes with references; long table spanning pages; figure with caption placed visually below but logically referenced above (both patterns exist and must be handled intentionally). [\[32\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)

**Topic: artifacts vs real content**

1.  **Definition.** An artifact is decoration/pagination/layout content not intended to be relevant for understanding; it is typically ignored by AT. Real content is the meaningful document content that must be semantically represented in the structure tree. PDF/UA-1 and Matterhorn emphasize correct classification; WTPDF in PDF 2.0 adds richer artifact structure typing/attributes and clarifies when artifact property list entries vs Artifact structure elements are appropriate. [\[33\]](https://pdfa.org/glossary-of-accessibility-terminology-in-pdf/)
2.  **Why it matters for a PDF Accessibility Expert.** Misclassifying artifacts causes either (a) noise and confusion when decorative content is read, or (b) loss of meaning when real content is hidden as artifact. Matterhorn explicitly defines both directions as failure conditions. [\[34\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)
3.  **Default rules.** Mark as artifacts: decorative rules, nonsemantic spacing, background imagery, table grid lines used purely for layout, repeated running headers/footers, page numbers, watermarks, redaction indicators, etc.; ensure semantic content is not placed inside artifact-marked content sequences; in PDF 1.7 contexts, artifacts are typically represented via marked-content artifact property lists (e.g., pagination), and best practice guidance calls out page numbers as artifacts and encourages Page Labels for accessible enumeration; in PDF 2.0 contexts, use the Artifact structure element + semantically significant attributes (Subtype such as Header/Footer/Watermark/PageNum/Bates/LineNum/Redaction) where appropriate. [\[35\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
4.  **Exception rules.** If what looks like a header/footer/watermark contains **unique** legally or contextually required information (not merely repetitive pagination), treat it as real content *or* provide an explicit accessible alternative (e.g., a Note or dedicated section) while still maintaining a clean main reading order. When an artifact is meaningful only in the context of structure-tree content, WTPDF requires it be enclosed in an Artifact structure element (i.e., the artifact becomes a semantic adjunct rather than free-floating layout noise). [\[36\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)
5.  **Fallback logic.** When uncertain, prefer conservative inclusion (treat as real content) only if it appears meaning-bearing; otherwise treat as artifact—but log an uncertainty and force a human checkpoint for that element class (headers/footers/watermarks are common ambiguity zones). Use repetition detection across pages and “proximity to margins” heuristics as the first-pass classifier. [\[37\]](https://pdfa.org/glossary-of-accessibility-terminology-in-pdf/)
6.  **Failure conditions.** Artifact tagged as real content; real content marked as artifact; artifact content inside tagged content or tagged content inside artifact-marked sequences; semantic content embedded in artifacts (explicitly called semantically inappropriate by best practice guidance). [\[38\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)
7.  **Measurable thresholds.** Required: 0 instances of semantic content nested in artifact containers; 0 Matterhorn-style misclassification failures for PDF/UA-1 conformance claims. Recommended: for each repeated artifact class (header/footer/page number), ensure 100% consistency of classification across pages. [\[39\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)
8.  **Implementation guidance for an AI operator pack.** Implement an “ArtifactPolicy” module with deterministic classification tiers (decorative, pagination, watermark, redaction indicator, layout-only table lines) and representation rules that vary by target profile (PDF/UA-1 vs PDF/UA-2/WTPDF). Include an “artifact containment checker” that ensures artifacts are not simultaneously treated as real content (“cannot be both” is a common implementer framing in Tagged PDF education materials). [\[40\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)
9.  **Test cases.** Running header with document title repeated; footer with confidentiality clause that changes by section; watermark “DRAFT”; Bates numbering; a page number whose visible value does not match Page Labels; a table with visible gridlines and no actual semantic need for them. [\[41\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)

**Topic: alt text and figure treatment**

1.  **Definition.** Figures are non-text graphics objects representing content; PDF/UA-1 requires graphics objects (other than text objects) be tagged as Figure unless they are artifacts, and Figure tags must include an alternative representation or replacement text. WTPDF (aligned with PDF/UA-2-level accessibility) requires each Figure structure element to have at least one of Alt or ActualText. [\[42\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
2.  **Why it matters for a PDF Accessibility Expert.** Screen reader users need meaningful substitutes for non-text content, and structured figure handling also affects reuse/extraction pipelines (e.g., when figures contain text, or when figures are grouped). Requirements extend beyond images: link annotations require alternate descriptions, and figure+caption relationships influence comprehension. [\[43\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
3.  **Default rules.** For meaningful figures: tag as \<Figure\> and provide Alt (brief description) and/or ActualText (replacement text for content that is “textual” in intent); include captions using \<Caption\>; group multiple graphics with combined semantic value under a single Figure tag; prefer real text objects instead of text-as-image when possible; keep alt text concise (tool guidance), and avoid adding Alt on a parent tag that has child tags because doing so can cause screen readers to skip child content. [\[44\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
4.  **Exception rules.** Decorative/background images should be artifacts (no alt), including backgrounds behind link annotations where the link’s alternate text must cover both the graphic and the link meaning in PDF/UA-1 guidance. For complex figures (charts/diagrams), WTPDF explicitly contemplates substructure to describe subunits; in PDF 2.0 contexts, consider associated files (e.g., a CSV or long description) as a robust fallback for complex data graphics when Alt alone cannot convey meaning. [\[45\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
5.  **Fallback logic.** If you cannot generate sufficient alt text automatically with high confidence, generate a minimal “purpose-level” Alt and flag for human rewrite; if the figure is data-heavy, attach an accessible alternative (associated file) and link to it from surrounding content to preserve usability. [\[46\]](https://pdfa.org/iso-14289-2-pdfua-2/)
6.  **Failure conditions.** Missing Alt/ActualText for Figure (WTPDF accessibility conformance requirement); misleading Alt; Alt set on a container tag causing suppression of child reading; for annotations, mismatch between a structure element’s Alt and an annotation’s Contents when both are used for alternate descriptions. [\[47\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)
7.  **Measurable thresholds.** Required: 100% of meaningful figures have Alt or ActualText; 0 decorative images in the real-content tag tree; 0 Alt/Contents mismatches for annotations when both are present. Recommended: for alt quality, enforce a human-review sample (e.g., review all figures if ≤20; otherwise sample ≥10 + all “complex figure” detections). [\[48\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)
8.  **Implementation guidance for an AI operator pack.** Implement `FigurePolicy`: (a) classify figure as decorative vs meaningful vs “textual graphic”; (b) choose Alt vs ActualText vs both; (c) ensure placement in semantically appropriate block-level container (WTPDF notes require Figure elements with ActualText be contained within semantically appropriate block-level elements); (d) enforce “Alt only on leaf tags” to avoid screen reader suppression of children. [\[49\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)
9.  **Test cases.** Decorative flourish line; chart with labeled axes + legend; screenshot of an error dialog (UI screenshot) requiring text equivalent; logo encoded as text with missing Unicode mapping; figure split across pages; figure inside a link. [\[50\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)

**Topic: table structure and header association**

1.  **Definition.** Accessible tables must be represented using table structure elements (\<Table\>, \<TR\>, \<TH\>, \<TD\>, optionally \<THead\>/\<TBody\>/\<TFoot\>), and headers must be programmatically associated with the data cells (via Scope and/or IDs+Headers mechanisms). Tagged PDF best practices and WTPDF provide concrete patterns and attribute expectations. [\[51\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
2.  **Why it matters for a PDF Accessibility Expert.** Screen readers announce table structure, position, and header context; without header association, users hear disconnected values. Tooling guidance and remediation manuals directly call out missing header/data relationships as a primary accessibility failure mode. [\[52\]](https://www.newformat.se/documents/axes4-sw/axespdf-docs/newformat-ab-sweden-axesPDF_Guide_EN_User_Guide_V07-2021-2021-07-29.pdf)
3.  **Default rules.** Use real table tags only when meaning depends on a row/column matrix (never use table tags for layout alignment); ensure each row uses \<TR\>; cells are \<TH\> or \<TD\>; empty cells are always \<TD\> (not \<TH\>); spanning cells include correct RowSpan/ColSpan; for multipage tables, represent as a single logical Table element; repeated visual header rows on subsequent pages should be artifacts (avoid duplicate reading) while the semantic header row remains in the logical table; use Scope wherever possible and Headers+ID where algorithmic header determination is insufficient (complex tables). [\[53\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
4.  **Exception rules.** Irregular tables are not strictly prohibited under PDF/UA-1, but Matterhorn notes they are strong indicators of improper structure, so treat them as “warn → human review required.” For tables whose visual form is fundamentally incompatible with a navigable matrix (e.g., heavily merged cells for layout), consider restructuring content into lists or separate smaller tables rather than perpetuating an unusable table model. [\[54\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)
5.  **Fallback logic.** If tagging is present but header association is missing, attempt automatic inference: detect header rows/columns (position, typography, repeated patterns), assign Scope for simple cases, and fall back to explicit IDs+Headers mapping for complex cases; if inference confidence is low, flag for manual table modeling (Acrobat guidance notes cell-level table tagging is labor-intensive and often better fixed in the authoring application). [\[55\]](https://www.newformat.se/documents/axes4-sw/axespdf-docs/newformat-ab-sweden-axesPDF_Guide_EN_User_Guide_V07-2021-2021-07-29.pdf)
6.  **Failure conditions.** Layout content incorrectly tagged as a table; missing \<TR\> structure; \<TH\> with no associated subcells; Scope missing or inconsistent; Headers references not pointing to \<TH\>; repeated header rows treated as real content duplicates; empty header cells; incorrect RowSpan/ColSpan for merged cells. [\[56\]](https://www.newformat.se/documents/axes4-sw/axespdf-docs/newformat-ab-sweden-axesPDF_Guide_EN_User_Guide_V07-2021-2021-07-29.pdf)
7.  **Measurable thresholds.** Required: 0 layout tables tagged as \<Table\>; 100% \<TH\> cells have valid associations (Scope and/or Headers); 100% merged cells have correct span attributes. Recommended: run a “screen reader table navigation simulation” on at least one complex table per document (or all tables if ≤5) to confirm headers are announced as expected. [\[52\]](https://www.newformat.se/documents/axes4-sw/axespdf-docs/newformat-ab-sweden-axesPDF_Guide_EN_User_Guide_V07-2021-2021-07-29.pdf)
8.  **Implementation guidance for an AI operator pack.** Implement `TableModeler`: (a) detect tables; (b) build canonical matrix model; (c) map to tag structure; (d) compute header association using Scope when possible; (e) generate IDs+Headers where necessary; (f) artifact repeated visual headers; (g) validate with rules from Tagged PDF best practice examples and WTPDF table attribute expectations (RowSpan/ColSpan/Headers/Scope/Summary). [\[51\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
9.  **Test cases.** Simple 2×2 table with header row; complex table with multi-level column headers; multipage table with repeated header row; “table used for layout” (logos aligned with text); table with blank spacer column; table with merged totals row (semantic border meaning). [\[57\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)

**Topic: Unicode mapping and extraction fidelity**

1.  **Definition.** Unicode mapping is the mechanism by which text in a PDF (page content, metadata, annotations) is mapped to Unicode code points so processors can reliably extract, search, copy/paste, and present the text through AT; in PDF 2.0 contexts, WTPDF adds constraints on ToUnicode mappings and governs replacement mechanisms (ActualText) when the natural mapping is inadequate. [\[58\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
2.  **Why it matters for a PDF Accessibility Expert.** Unicode mapping is not only a screen reader requirement; it is the foundation for braille output, text reuse, reliable search, and any downstream NLP/LLM extraction. Both standards and best practices treat missing/incorrect mapping as accessibility-breaking. [\[59\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
3.  **Default rules.** Ensure all text maps to Unicode; prefer correct ToUnicode CMaps; where symbolic fonts or nonstandard glyphs are used, provide Unicode via ToUnicode or ActualText; discourage Unicode PUA usage (and if used, require ActualText/Alt); ensure language is declared and language changes identified; in PDF 2.0 contexts, enforce that ToUnicode values are \>0 and not U+FEFF or U+FFFE (WTPDF requirement). [\[60\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
4.  **Exception rules.** When authors use text-like graphics (logos, stylized text, stretchable glyph constructions), standards and glossary guidance point to ActualText as a corrective mechanism; however, in PDF 1.7 contexts ActualText can affect what AT announces about tag types, so best practice is to place ActualText on a Span (or Span marked-content sequence) rather than on higher-level structural tags when you do not intend to replace semantics. [\[61\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
5.  **Fallback logic.** If ToUnicode cannot be generated or repaired for specific content, apply ActualText at the smallest semantically safe scope (Span-level), and then validate extraction via tool-based “save as accessible text” or equivalent extraction tests; if extraction remains unreliable, treat it as a stop condition requiring re-authoring or deeper font-level repair. [\[62\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
6.  **Failure conditions.** Missing Unicode mapping; ToUnicode maps to invalid values (0, FEFF, FFFE); symbolic glyphs without a Unicode/ActualText substitute; reliance on Unicode PUA without ActualText/Alt; language not declared or language changes not tagged. [\[63\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)
7.  **Measurable thresholds.** Required: 0 unmapped characters in real content; 0 invalid ToUnicode mappings; 100% of symbolic/textual nonstandard glyphs have a meaningful Unicode substitute. Recommended internal KPI: “round-trip extraction fidelity”—extract text in logical reading order and require ≥99.5% character-level match against a ground-truth transcript for documents intended for reuse; note that PDF/UA does not mandate a numeric threshold, but it *does* require correct mapping and correctable OCR. [\[64\]](https://cdn.standards.iteh.ai/samples/82278/61329d7c9eb54e3fbc58a69f6036031d/ISO-14289-2-2024.pdf)
8.  **Implementation guidance for an AI operator pack.** Implement `UnicodeFidelity`: (a) inspect all fonts and verify ToUnicode presence/validity; (b) detect PUA usage and enforce ActualText/Alt; (c) detect symbolic fonts (e.g., Wingdings-like) and map; (d) enforce Span-level ActualText strategy in PDF 1.7 contexts; (e) run extraction-based regression tests (logical order extraction) and record diffs as measurable outputs. [\[65\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
9.  **Test cases.** PDF with embedded symbolic font bullets; multilingual paragraph with language switch; text where copy/paste produces gibberish; ToUnicode mapping that yields U+FEFF; PUA-mapped glyph without ActualText; metadata title containing non-Unicode-mapped characters. [\[66\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)

**Topic: OCR fallback and remediation**

1.  **Definition.** OCR remediation converts raster-only (image-only) pages into machine-readable text, typically by adding an invisible text layer aligned to the image, and then tagging that text so it participates in the logical structure. Standards explicitly state that OCR-generated errors must be corrected and the content tagged. [\[67\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
2.  **Why it matters for a PDF Accessibility Expert.** Image-only PDFs are inaccessible to screen readers by default; OCR without correction can produce incorrect content that is actively harmful (wrong numbers, wrong instructions). The PDF Association glossary explicitly states that OCR accuracy is critical and users performing text recognition have an obligation to ensure the final result is correct. [\[68\]](https://helpx.adobe.com/acrobat/using/create-verify-pdf-accessibility.html)
3.  **Default rules.** Detect image-only pages (no fonts/text objects) and run OCR in the correct language; create machine-readable text; correct OCR errors; tag the resulting text into proper structure with correct reading order; preserve real images as figures with Alt where needed; use ActualText to correct localized OCR errors or to replace content intended to be consumed as text when correct text is otherwise unavailable. [\[69\]](https://helpx.adobe.com/acrobat/using/create-verify-pdf-accessibility.html)
4.  **Exception rules.** For handwriting, poor scans, or degraded originals, OCR confidence may be too low for safe automated remediation; for these cases, the strict expert must escalate to human correction or provide an accessible alternative representation. ActualText can correct errors, but it must be applied in a way that does not unintentionally suppress semantics (Span-first best practice in PDF 1.7 contexts). [\[70\]](https://pdfa.org/glossary-of-accessibility-terminology-in-pdf/)
5.  **Fallback logic.** If OCR quality is below internal thresholds (see below), do not ship as “accessible” solely because tags exist; produce a remediation report marking the document “requires manual OCR correction,” and optionally generate an auxiliary accessible text artifact (TXT/HTML) for immediate access while remediation is pending. [\[71\]](https://pdfa.org/glossary-of-accessibility-terminology-in-pdf/)
6.  **Failure conditions.** Image-only PDF left un-OCR’d; OCR text misordered relative to layout; OCR errors uncorrected; OCR text not tagged or incorrectly tagged; OCR causing incorrect Unicode mapping; invisible text layer not aligned to correct reading sequence (common in multi-column scans). [\[72\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
7.  **Measurable thresholds.** Required (normative): OCR-generated errors must be corrected and content tagged. Recommended internal thresholds: (a) for each document, sample ≥200 words across at least 3 pages; require ≥99% word correctness for general prose; raise to ≥99.5% for numeric/legal/regulatory documents; (b) 0 critical-field errors (dates, money, totals) in sampled regions; (c) 0 “garbled” Unicode replacement patterns after extraction. These numbers are an engineering policy layer; PDF/UA itself defines the requirement to correct errors, not a numeric rate. [\[73\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
8.  **Implementation guidance for an AI operator pack.** Create an `OCRPipeline`: detect image-only; select OCR language; run OCR; post-correct with dictionaries + layout constraints; reconcile OCR text with tagging (block segmentation and reading order); add ActualText patches for residual localized errors; run extraction regression tests and flag low-confidence regions for human review. [\[74\]](https://helpx.adobe.com/acrobat/using/create-verify-pdf-accessibility.html)
9.  **Test cases.** Skewed scan; low-resolution photocopy; multi-column scanned paper; scanned table with gridlines; scanned page with a chart containing labels; mixed-content PDF where some pages are text and some are raster. [\[75\]](https://pdfa.org/glossary-of-accessibility-terminology-in-pdf/)

**Topic: ligatures, punctuation, and copy-paste integrity**

1.  **Definition.** Ligatures (e.g., “fi”, “fl”) and certain punctuation/typographic constructs may be encoded as single glyphs or multi-glyph constructions that do not map cleanly to Unicode unless explicitly mapped; PDF/UA-1 even calls out “stretchable characters” (e.g., parentheses/brackets constructed from multiple glyphs) as requiring ActualText. Tagged PDF best practices recommend ToUnicode where possible and ActualText for cases like associating a ligature with a sequence of separate characters. [\[76\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
2.  **Why it matters for a PDF Accessibility Expert.** If extraction yields missing letters or broken punctuation, screen readers may mispronounce content, braille displays may show blanks, and copy/paste can corrupt data (e.g., document numbers, medical terms, contract clauses). These are “silent” failures that visual QA often misses, so a strict expert must treat extraction integrity as a first-class requirement. [\[77\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
3.  **Default rules.** Provide ToUnicode mapping for ligature glyphs (mapping to the correct Unicode sequence) whenever possible; otherwise apply ActualText at Span-level to provide the intended character sequence; apply ActualText for stretchable punctuation constructions as required; avoid Unicode PUA as a semantic crutch; treat bullets and symbolic characters similarly (ToUnicode or ActualText). [\[78\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
4.  **Exception rules.** If a ligature is purely decorative but still represents alphabetic content, it is not a semantic exception—normalize it for extraction; the only real “exception” class is content not intended to be consumed as text (then it should be an artifact or a properly-described figure), but that is a classification decision, not a Unicode mapping exemption. [\[79\]](https://pdfa.org/glossary-of-accessibility-terminology-in-pdf/)
5.  **Fallback logic.** If ligature/punctuation extraction failures are widespread, do not patch piecemeal; instead repair at the font/ToUnicode mapping level (preferred) or re-embed fonts with valid mappings if you control generation. If you only control remediation, apply systematic Span-level ActualText in a deterministic pattern and rerun extraction regression tests. [\[80\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
6.  **Failure conditions.** Copy/paste drops characters (nulls, blanks), merges words incorrectly (ligature not expanded), or produces incorrect punctuation; ActualText applied at the wrong semantic level causing AT to lose tag-type announcements (PDF 1.7 best practice warning); PUA use without ActualText/Alt; “stretchable characters” not represented via ActualText. [\[81\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
7.  **Measurable thresholds.** Required: zero missing characters in extracted text for real content; enforce a “ligature suite” test (“office”, “affiliate”, “shelf”, “efficient”, etc.) and require exact match; for punctuation, enforce exact match on bracketed expressions, citations, and formula-like strings. These are internal measurable policies built on the standards’ requirement for correct Unicode representation. [\[76\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
8.  **Implementation guidance for an AI operator pack.** Build `ExtractionFidelityHarness`: select test strings (or sample spans), extract text in logical order, compare against expected; if mismatch, classify root cause (ToUnicode missing/invalid vs ligature glyph vs constructed punctuation vs OCR error); apply targeted fix (ToUnicode repair → ActualText Span patch → OCR correction). Maintain a regression baseline per document template family. [\[82\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
9.  **Test cases.** Ligatures in small caps; bullet lists using symbolic glyphs; brackets constructed from multiple glyphs; a PDF where soft hyphens appear in extracted text; content with many em-dashes/ellipsis; mixed fonts with different ToUnicode behavior. [\[83\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)

**Topic: document vs UI accessibility differences**

1.  **Definition.** Document accessibility is about semantic structure and content representation (tag tree, reading order, Unicode, alt text). UI accessibility in PDFs concerns interactive constructs such as annotations (links, comments, markups), forms (widget annotations/fields), keyboard navigation (tab order), and security settings that can block text extraction required by screen readers. In PDFs, annotations exist separately from page content but are considered content for accessibility and must be represented appropriately. [\[84\]](https://pdfa.org/glossary-of-accessibility-terminology-in-pdf/)
2.  **Why it matters for a PDF Accessibility Expert.** A PDF may be “tagged” but still fail users if links lack accessible names, tab order is wrong, form fields aren’t in the structure tree, or security settings prevent screen readers from extracting text. Tool guidance explicitly flags these as accessibility issues and provides remediation steps. [\[85\]](https://helpx.adobe.com/acrobat/using/create-verify-pdf-accessibility.html)
3.  **Default rules.** Ensure interactive elements are integrated with structure: link annotations must be nested in \<Link\> structure elements; provide alternate descriptions for link annotations; ensure tab order uses document structure on pages with annotations; ensure accessibility permissions/security do not block AT’s ability to copy/extract text; for annotations where both Contents and enclosing structure Alt are used as alternative descriptions, WTPDF requires them to be identical. [\[86\]](https://www.newformat.se/documents/axes4-sw/axespdf-docs/newformat-ab-sweden-axesPDF_Guide_EN_User_Guide_V07-2021-2021-07-29.pdf)
4.  **Exception rules.** Some PDF features are deprecated in PDF 2.0 (e.g., XFA), so a strict expert should avoid relying on those constructs when targeting PDF/UA-2 workflows; when a feature cannot be made accessible in a given PDF processor ecosystem, the correct response is to provide an accessible alternative or to re-author the document. [\[87\]](https://cdn.standards.iteh.ai/samples/75839/b0970522e8464fffa9081aa71b03ca6c/ISO-32000-2-2020.pdf)
5.  **Fallback logic.** If an automated checker flags “Needs manual check,” treat it as a required human checkpoint rather than as a pass; use PAC’s screen reader/structure preview or equivalent to manually confirm navigation and reading experience when UI elements are present. [\[88\]](https://helpx.adobe.com/acrobat/using/create-verify-pdf-accessibility.html)
6.  **Failure conditions.** Untagged annotations or links not in \<Link\> tags; tab order not aligned with structure; missing alternate descriptions for links; Alt/Contents mismatch; security settings preventing AT extraction; form widgets not referenced correctly from the structure tree. [\[89\]](https://helpx.adobe.com/acrobat/using/create-verify-pdf-accessibility.html)
7.  **Measurable thresholds.** Required: 100% of interactive elements are reachable in logical navigation order via keyboard; tab order is structure-based for all pages with annotations; 0 security blocks for AT text extraction; WTPDF’s Alt/Contents equality holds wherever applicable. [\[90\]](https://helpx.adobe.com/acrobat/using/create-verify-pdf-accessibility.html)
8.  **Implementation guidance for an AI operator pack.** Implement `InteractiveAccessibility`: enumerate annotations/widgets; ensure each has structural representation and accessible name/description; enforce tab order policy; validate security/accessibility permission flags; create test scripts for Tab traversal and link-name announcements (AT simulation). [\[91\]](https://helpx.adobe.com/acrobat/using/create-verify-pdf-accessibility.html)
9.  **Test cases.** Link over an image with both Alt and Contents present; multi-page form with inconsistent tab order; highlight annotation spanning pages; comments added after tagging; PDF with restricted copying that blocks some screen readers. [\[92\]](https://helpx.adobe.com/acrobat/using/create-verify-pdf-accessibility.html)

**Topic: failure modes in accessible PDF editing**

1.  **Definition.** Accessible PDF editing failures occur when visual edits (text changes, object moves, optimization/flattening) are performed without preserving or updating the logical structure (tag tree, marked-content associations, role maps, reading order, metadata, Unicode mapping). Multiple remediation guides warn that editing can corrupt surrounding tag structure and necessitates rechecking tags after any edits. [\[15\]](https://highways.dot.gov/sites/fhwa.dot.gov/files/had21001.pdf)
2.  **Why it matters for a PDF Accessibility Expert.** Production systems that “round-trip” PDFs through tools or libraries frequently break tags, reorder structure, or drop accessibility metadata—resulting in regressions that can be subtle and hard to detect without systematic validation and extraction tests. [\[93\]](https://highways.dot.gov/sites/fhwa.dot.gov/files/had21001.pdf)
3.  **Default rules.** Treat any edit as a semantic operation: after edits, revalidate tagging, reading order, table structure, Unicode mapping, metadata, and interactive accessibility; avoid tool workflows known to destabilize tags (e.g., visual “Edit PDF” operations) unless followed by a full remediation retest; do not use reading-order UI tools as the source of truth when they disagree with tag structure (guidance warns they can cause additional tag issues and recommends prioritizing tag structure). [\[94\]](https://highways.dot.gov/sites/fhwa.dot.gov/files/had21001.pdf)
4.  **Exception rules.** Pure metadata edits (e.g., setting dc:title, DisplayDocTitle) can be applied without retagging content, but still require validation because metadata is itself a conformance requirement in PDF/UA-1; similarly, small text edits that do not change segmentation may still break Unicode mapping if fonts or ToUnicode entries are altered, so they are not automatically “safe.” [\[95\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
5.  **Fallback logic.** If an edit operation would require widespread retagging or risks breaking the structure tree (e.g., flattening/rasterization workflows, heavy optimization), prefer re-authoring from the source document and re-exporting with tags; if not possible, accept that a full “rebuild tag tree” remediation may be required and schedule it explicitly rather than attempting partial fixes. [\[96\]](https://highways.dot.gov/sites/fhwa.dot.gov/files/had21001.pdf)
6.  **Failure conditions.** Post-edit: untagged content introduced; artifacts reclassified incorrectly; reading order drift; tables lose header associations; ToUnicode mappings become invalid; title/display settings regress; tool-driven tag tree corruption. [\[97\]](https://highways.dot.gov/sites/fhwa.dot.gov/files/had21001.pdf)
7.  **Measurable thresholds.** Required: no new validation failures compared to pre-edit baseline; pass target validation profiles; 0 extraction regressions for sampled text; 0 reading-order regressions for sampled transitions; 0 table-navigation regressions for sampled tables. [\[98\]](https://docs.verapdf.org/validation/)
8.  **Implementation guidance for an AI operator pack.** Build `SemanticEditGuard`: before edit, snapshot (a) structure tree hash, (b) per-page content-to-tag linkage stats, (c) extracted text baseline, (d) table header model, (e) metadata conformance fields; after edit, recompute and diff; if diff exceeds allowed bounds, block release and route to remediation pass. [\[99\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)
9.  **Test cases.** Insert a paragraph; replace a word containing ligatures; edit a table cell; merge two tagged PDFs; add a watermark; run “optimize” or equivalent; add a link annotation after tagging; change document title. [\[100\]](https://highways.dot.gov/sites/fhwa.dot.gov/files/had21001.pdf)

## Implementation blueprint for a modular AI design operator pack

A modular operator pack should be designed as a pipeline of explicit analyzers and repair operators, each with deterministic inputs/outputs, severity scoring, and validation checkpoints. Two design constraints dominate:

- **The pack must not rely on “visual-only” cues**; it must operate on the PDF’s semantic and structural representation (including structure tree, marked-content sequences, and Unicode mapping mechanisms). [\[101\]](https://pdfa.org/wp-content/uploads/2012/07/Taking-Tagged-PDF-To-The-Next-Level.pdf)
- **The pack must remain “semantics-preserving” under edits**, meaning every change is tracked both in rendered content and in accessibility semantics; otherwise you cannot safely ship “patched PDFs” at scale. [\[96\]](https://highways.dot.gov/sites/fhwa.dot.gov/files/had21001.pdf)

**Recommended module set (operator pack architecture):**

**Ingest and classify** - `DetectTargetProfile`: determine whether the target is PDF/UA-1 (PDF 1.7) or PDF/UA-2/WTPDF (PDF 2.0); enforce profile-specific rules (e.g., artifact representation differences; deprecated features in PDF 2.0). [\[102\]](https://pdfa.org/resource/iso-14289-pdfua/)  
- `ContentAccounting`: ensure all content is either real content tagged, artifacted, or removed from accessibility structure. [\[11\]](https://helpx.adobe.com/acrobat/using/editing-document-structure-content-tags.html)

**Structure and semantics** - `StructTreeValidator`: verify presence and coherence of logical structure; role mapping; semantic appropriateness; containment rules (especially under PDF 2.0 + ISO/TS 32005). [\[103\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)  
- `ReadingOrderResolver`: ensure depth-first structure traversal yields correct reading order; handle multipage elements; enforce tag-order truth over paint order. [\[104\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)

**Media and alternates** - `FigureAltGenerator`: generate and attach Alt/ActualText as required; enforce leaf-tag alt placement; handle complex figures via substructure or associated files in PDF 2.0 contexts. [\[105\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)  
- `ArtifactClassifier`: detect and correctly represent artifacts including pagination subtypes (especially when targeting WTPDF/PDF 2.0). [\[106\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)

**Text fidelity** - `UnicodeMapper`: validate ToUnicode/ActualText; enforce restrictions; flag PUA; apply Span-level ActualText fixes. [\[107\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)  
- `ExtractionFidelityHarness`: regression-test copy/paste/text extraction in logical order and compare to expected results (document templates can maintain ground-truth strings). [\[108\]](https://helpx.adobe.com/acrobat/using/create-verify-pdf-accessibility.html)

**Tables** - `TableModeler`: build and validate table structure + header associations; detect and reject layout tables; enforce Scope/Headers behavior. [\[109\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)

**OCR** - `OCRPipeline`: detect image-only pages; OCR with language; correct errors; then tag and order; require human review on low-confidence regions. [\[110\]](https://helpx.adobe.com/acrobat/using/create-verify-pdf-accessibility.html)

**Metadata and conformance claims** - `MetadataEnforcer`: ensure required title/DisplayDocTitle settings for PDF/UA-1 and correct identification schema for PDF/UA-2 claims (e.g., XMP identification). [\[111\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)

**Validation and reporting** - `MachineValidation`: run veraPDF (PDF/UA-1, PDF/UA-2, WTPDF profiles as appropriate); record all machine failures. [\[112\]](https://docs.verapdf.org/validation/)  
- `HumanCheckScheduler`: map Matterhorn “Human” checkpoints + known non-machine-verifiable requirements (reading order correctness, alt text adequacy) into required manual tests; integrate PAC structure/screen-reader preview as support. [\[113\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)

## Validation methodology and measurable thresholds

A strict operator pack should implement a **two-layer acceptance gate**:

**Layer one: machine-verifiable conformance** - Must pass the relevant validation profile(s) with **0 errors**: PDF/UA-1 or PDF/UA-2 (and WTPDF when targeting PDF/UA-2-level accessibility behavior). [\[112\]](https://docs.verapdf.org/validation/)  
- Must satisfy conformance-required metadata where applicable (e.g., PDF/UA-1 requirements for dc:title and DisplayDocTitle). [\[114\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)

**Layer two: human-verifiable semantic accessibility** - Must complete required human checks (Matterhorn explicitly distinguishes human judgment failure conditions; PAC explicitly supports manual/visual checking alongside automated checks). [\[115\]](https://pdfa.org/download-area/publications/Matterhorn-Protocol-1-1.pdf)  
- Must pass structured sampling checks for: reading order; table navigation; alt-text adequacy; OCR accuracy (when OCR was used); copy/paste correctness on selected strings. ISO and PDF Association guidance emphasize the requirement to correct OCR errors and the criticality of OCR accuracy, while PDF/UA’s scope limitations mean your organization must define numeric test thresholds. [\[116\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)

**Tooling triangulation (recommended):**  
Because validators can differ in coverage and interpretation, triage should use at least one open validator (veraPDF) and one end-user-focused checker/preview tool (PAC) for both automated checks and manual structure preview. [\[117\]](https://docs.verapdf.org/validation/)

**A. Condensed operating spec**

- **Inputs:** PDF bytes + declared target (PDF/UA-1 vs PDF/UA-2/WTPDF) + language(s) + release intent (“publish as accessible PDF”). [\[118\]](https://pdfa.org/resource/iso-14289-pdfua/)

- Unicode mapping is correct and extraction-safe; use ToUnicode first; ActualText as precise fallback (Span-level best practice in PDF 1.7 contexts). [\[122\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)

- Verify multipage elements (paragraphs, tables) remain single logical structures (no per-page restarts). [\[121\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
- Verify artifacts (headers/footers/page numbers) are not interleaved into main reading order unless intentionally modeled as contextual artifacts. [\[132\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
- Verify tab order uses document structure where annotations/forms exist. [\[133\]](https://www.hhs.gov/sites/default/files/pdf-tagging.pdf)
- Use structure/screen-reader preview to confirm order on at least: one multi-column region, one table, one figure+caption pairing. [\[134\]](https://pdfa.org/product/pac/)

**D. Extraction-safety checklist**

- Confirm ToUnicode mappings exist and are valid; ensure no invalid mapped values (e.g., prohibited values called out in WTPDF). [\[135\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)
- Confirm no reliance on Unicode PUA without ActualText/Alt. [\[136\]](https://pdfa.org/wp-content/uploads/2024/02/Well-Tagged-PDF-WTPDF-1.0.pdf)
- Confirm ligatures and symbolic glyphs extract correctly (use ToUnicode or Span-level ActualText). [\[137\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
- Confirm OCR text (if used) is accurate and corrected; validate with sampling and extraction regression tests. [\[138\]](https://cdn.standards.iteh.ai/samples/64599/f43fa811a07343258173535d830fed18/ISO-14289-1-2014.pdf)
- Confirm copy/paste on representative samples matches expected text exactly (especially numbers, headings, and table cells). [\[139\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)

1.  “I have a two-column PDF brochure exported from a design tool; it’s tagged but the screen reader jumps between columns. Produce a reordering plan that preserves semantic grouping and validates via structure traversal.” [\[140\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
2.  “This PDF has running headers with the chapter title and page numbers; decide what becomes artifact vs real content and how a user still learns their position in the document.” [\[141\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
3.  “A table spans 5 pages with repeated header rows on each page; repair the tags so the table is one logical \<Table\> and repeated visual headers do not cause duplicate reading.” [\[142\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
4.  “This PDF uses a table purely to align logos and contact info; remove table semantics and replace with correct structural tags while preserving layout.” [\[143\]](https://pdfa.org/wp-content/uploads/2019/06/TaggedPDFBestPracticeGuideSyntax.pdf)
5.  “A complex table has multi-level headers (row + column headers). Create IDs+Headers associations and specify where Scope is sufficient vs insufficient.” [\[144\]](https://www.newformat.se/documents/axes4-sw/axespdf-docs/newformat-ab-sweden-axesPDF_Guide_EN_User_Guide_V07-2021-2021-07-29.pdf)
