# Writing and Case Study Summary

## Purpose
Canonical summary for UX copy, case-study structure, documentation clarity, audit/rebuild/expand mode control, design rationale, and evidence framing.

## Use when
- the task is a rewrite, audit, rebuild, simplification, CTA/error message, explanatory note, documentation pass, or case-study narrative

## Default rules
- identify the writing mode first: case study, audit, rebuild plan, expand proposal, UX copy, or documentation
- lead with the user task, problem, or key point
- use short sentences and active voice
- in case studies, keep the backbone visible: problem → process → solution → outcome
- tie design claims to rationale and evidence instead of adjectives
- annotate visuals so they support the narrative instead of floating as artifacts
- do not mix portfolio, documentation, and UX microcopy conventions blindly

## Useful thresholds
- aim below roughly 20 to 25 words per sentence when possible
- one main idea per paragraph
- case studies should foreground role, problem framing, decisions, and outcomes
- UX microcopy should stay extremely brief and action-oriented
- documentation should use headings and steps rather than narrative drift
- if evidence is unavailable, say so instead of implying proof

## Writing-mode matrix
- Case Study: professional narrative, first person acceptable, decision/outcome centered
- Audit: issue → impact → recommendation
- Rebuild: direct, ordered, imperative, implementation-oriented
- Expand: current state → proposed additions
- UX Copy: short, direct, task-focused
- Documentation: neutral, explanatory, structured

## Failure patterns
- writing a rebuild answer like an audit
- writing a case study like marketing copy
- error messages with no next step
- design claims with no evidence
- buried benefits, missing problem framing, or outcomes hidden at the end
- screenshots or artifacts with no caption or narrative role
