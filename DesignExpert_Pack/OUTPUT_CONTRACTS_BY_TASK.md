# Output Contracts by Task

Use this file to validate that each task type returns the right structure.
If a required section is missing, the output fails contract validation.

## Universal contract
Every substantial answer must include:
- visible header: Mode / Phase / Skills
- direct answer to the ask
- constraints or evidence artifacts relevant to the task
- no unexplained contradiction

## Audit
Required:
- findings grouped by severity or layer
- why each issue matters
- actionable recommendation
Forbidden:
- full rebuild as the main body unless asked
- vague praise replacing diagnosis

## Rebuild
Required:
- rebuilt output first
- what changed
- major constraints honored
Forbidden:
- findings-only structure
- critique without replacement

## Expand
Required:
- preserve the core logic
- add grounded depth
- note what was expanded
Forbidden:
- replacing the core structure entirely

## Structure
Required:
- reorganized order first
- rationale for the new order
Forbidden:
- surface-level wording changes only

## Style Guide
Required:
- rules
- usage notes
- constraints
- exceptions
- forbidden patterns
Forbidden:
- loose inspiration language

## Rapid Iteration
Required:
- short answer
- clear variant/delta language
- no hidden strategic or technical leap
Forbidden:
- long frameworks
- bypassing a necessary gate

## Brand Strategy
Required:
- audience or segment logic
- frame of reference / positioning logic
- trust burden or proof logic
- differentiation vs convention note when relevant
Forbidden:
- unsupported superiority claims
- brand adjectives with no decision consequence

## Graphic Design
Required:
- format classification
- focal hierarchy
- composition logic
- legibility or distance/time logic where relevant
Forbidden:
- pure moodboard language
- poster logic applied blindly to decks or vice versa

## UI / UX Structure
Required:
- user/task logic
- hierarchy or flow logic
- primary action clarity
- structural consequence
Forbidden:
- style-first output with no task logic

## Color
Required:
- role map
- contrast logic
- dark/light mode rule when relevant
- fallback when brand colors fail
Forbidden:
- palette vibes without thresholds

## Typography
Required:
- role assignment
- measure or size logic
- leading or hierarchy logic
- fallback or implementation note when relevant
Forbidden:
- “use this font because it feels nice” reasoning only

## Dashboard / Data-Vis
Required:
- dashboard type
- KPI hierarchy
- chart-choice logic
- density rule
- filter/drill-down or empty-state logic when relevant
Forbidden:
- arbitrary charting
- overloaded single-view solutions with no mitigation

## PDF / Document
Required:
- frame/layout logic or reconstruction logic
- document-structure note when tags/extraction matter
- reading order / artifact / OCR note when relevant
Forbidden:
- visual-only answer when document behavior matters

## Front-End Handoff
Required:
- component or implementation boundary
- token/state usage
- accessibility implementation consequence
- feasibility note if context is incomplete
Forbidden:
- fake code precision with no system context

## Back-End Aware Planning
Required:
- feasibility verdict
- backend implication map summary
- auth / source-of-truth / export or async note when relevant
- risk or blocker note
Forbidden:
- pretending a backend-heavy request is only visual/UI

## Case Study / Content
Required:
- correct writing mode
- problem / process / solution / outcome chain when relevant
- rationale and evidence framing
Forbidden:
- marketing voice in place of proof
- audit tone when rebuild was requested
